# Codebase Graph GitHub Publishing Design

## Goal

Publish `codebase-graph` as a public GitHub repository under `humeo`, add a repeatable GitHub release flow, and provide a one-line installation path based on `curl -fsSL ... | bash`.

## Current State

- The local repository has no Git remote configured.
- The package is already buildable as a Python CLI distribution:
  - Project name: `codebase-graph`
  - Version: `0.1.0`
  - Console entry point: `cg`
- Tests currently pass locally with `uv run pytest -v --tb=short`.
- Some local-only files exist and must not be published accidentally:
  - `.merge-backups/`
  - `.nvimlog`
  - `.pytest_cache/`
  - `.venv/`

## Chosen Approach

Use a public GitHub repository with versioned release artifacts and a bootstrap installer script.

This is preferred over a source-only installer because it gives a stable version boundary, lets users install an exact release, and makes the release page the canonical distribution point. It also avoids adding PyPI as a second publishing surface before the GitHub release flow is stable.

## Repository Design

- Create a public repository named `humeo/codebase-graph`.
- Add `origin` to the local repository and push `main`.
- Keep the first public version aligned with the package version already declared in `pyproject.toml`: `v0.1.0`.
- Publish only project files required for source, tests, documentation, workflows, and installation.
- Exclude local-only noise through `.gitignore` updates if needed.

## Release Design

### Release Trigger

Use a GitHub Actions workflow triggered by version tags matching `v*`.

### Release Steps

The workflow should:

1. Check out the repository.
2. Install `uv` using the official Astral installer.
3. Set up Python 3.12 to match the project requirement.
4. Run `uv sync`.
5. Run the full test suite.
6. Build release artifacts with `uv build`.
7. Create a GitHub release for the tag.
8. Upload the built `wheel` and `sdist` artifacts to that release.

### Release Assets

The initial release should publish:

- `dist/codebase_graph-0.1.0-py3-none-any.whl` or equivalent wheel name generated by the build backend
- `dist/codebase_graph-0.1.0.tar.gz` or equivalent source distribution name generated by the build backend

The exact filenames should not be hard-coded in the workflow beyond glob patterns, because hatchling determines the final names.

## Installer Design

### Entry Point

Expose a shell installer at:

- `https://raw.githubusercontent.com/humeo/codebase-graph/main/scripts/install.sh`

Primary install command:

```bash
curl -fsSL https://raw.githubusercontent.com/humeo/codebase-graph/main/scripts/install.sh | bash
```

### Installer Behavior

The script should:

1. Detect the current platform and fail clearly on unsupported platforms.
2. Ensure `curl` or `wget` is available.
3. Ensure `uv` is installed.
4. If `uv` is missing, bootstrap it using Astral's official installer.
5. Resolve the version to install:
   - default: latest GitHub release
   - optional: explicit version through an environment variable such as `CODEBASE_GRAPH_VERSION`
6. Install `codebase-graph` from the release artifact or release source archive using `uv tool install`.
7. Print a short success message including how to verify the install with `cg --version`.

### Version Resolution

The installer should prefer a release-specific artifact instead of installing from `main`.

That keeps the one-line install path tied to published releases and prevents silent drift between the documented install command and the actual code users receive.

### Update Behavior

If `cg` is already installed, the script should use the uv tool upgrade path rather than leaving duplicate state behind. The exact mechanism can be implemented as reinstall-or-upgrade, but the user-facing result must be idempotent.

## Documentation Design

Update the root `README.md` to include:

- public repository URL
- quick install command
- release model
- how to install a pinned version
- how to verify installation
- brief note that the installer bootstraps `uv` if required

Add or update docs to explain:

- how maintainers cut a release
- what the installer does
- how to install from source for development

## Git Hygiene

Before the first push:

- stage only intended project files
- avoid publishing `.merge-backups/`, `.nvimlog`, `.pytest_cache/`, `.venv/`, or other local state
- keep design/planning docs committed separately from implementation where practical

## Verification Plan

Before repository creation and release:

- run `uv run pytest -v --tb=short`
- run `uv build`
- execute the installer locally in a controlled form without `curl | bash`, e.g. `bash scripts/install.sh`

After repository creation:

- confirm `origin` points at `humeo/codebase-graph`
- confirm `main` is pushed
- confirm the release exists and contains build artifacts
- verify the documented install command works against the public repository

## Risks

- `curl | bash` only works cleanly for a public repository; private visibility would require a different auth-aware installer path.
- Release automation can drift from the package version if tags and `pyproject.toml` are not kept aligned.
- Installing from release assets is more stable than installing from `main`, but the script must still handle missing or malformed release metadata clearly.

## Out of Scope

- Publishing to PyPI
- Multi-platform standalone binaries
- Windows PowerShell bootstrap installer parity in the first pass
